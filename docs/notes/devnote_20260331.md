## 2. 아키텍처 변경 상세

### 2-1. 분류 엔진: if-else → 데이터 주도형

**이전 (하드코딩)**
```typescript
classify: (name, ext) => {
  const lc = name.toLowerCase();
  if (/월간보고|monthly/.test(lc)) return "00_업무보고\\월간보고";
  if (/임원/.test(lc)) return "00_업무보고\\임원회의";
  // ... 20줄 반복
  return "01_수신함";
}
```

**이후 (데이터 테이블)**
```typescript
interface ClassifyRule {
  pattern: string;     // 정규식 문자열
  folder: string;      // 대상 폴더 경로
  extMatch?: string;   // 확장자 카테고리 매칭
}

const RULES_DEVELOPER: ClassifyRule[] = [
  { pattern: "tc[-_]|테스트케이스|testcase|test.?case", folder: "30_QA\\32_테스트케이스" },
  // ... 68개 규칙
];

classify: (name, ext) => classifyByRules(RULES_DEVELOPER, name, ext, "01_수신함")
```

**장점:**
- 규칙 추가/수정이 1줄 단위로 가능 (코드 로직 변경 불필요)
- 향후 UI에서 사용자 커스텀 규칙 편집 가능
- JSON 파일로 팀 간 규칙 공유 가능
- 규칙별 매칭 통계 수집 가능

### 2-2. autoClassify 파일명 패턴 확장

| # | 패턴 | 정규식 | 예시 | 결과 |

### 2-5. 실행/되돌리기 시스템

**실행 흐름:**
```
템플릿 버튼 클릭 → 미리보기만 (폴더 미생성)
                    ↓
"N개 정리 실행" → 1) createFolders (템플릿+사용자지정)
                    2) organizeCustom (파일 이동, ops 반환)
                    3) 실행 로그 표시
                    ↓
완료 화면 → "↩ 되돌리기" 버튼 + 안내 배너
              ↓
되돌리기 → undoMoves IPC (ops 역순 renameSync)
```

**IPC 핸들러 3개 추가:**

| 핸들러 | 용도 |
|--------|------|
| `fs:findEmptyDirs` | 빈 폴더 재귀 탐색 |
| `fs:removeEmptyDirs` | 빈 폴더 삭제 |
| `fs:undoMoves` | 파일 이동 되돌리기 (ops 역순) |
